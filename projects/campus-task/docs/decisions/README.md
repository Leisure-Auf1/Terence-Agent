# 架构决策记录 (ADR)

## ADR-001: 为什么用 flat file 而非数据库

**状态**: ✅ 已采纳

**背景**: 实验 1 要求数据持久化，但该工具是教学演示性质，数据量小。

**决策**: 使用 JSON 文件 (`tasks.json`) 存储，而非 SQLite 或其他数据库。

**理由**:
- 零依赖 — 不需安装数据库驱动
- 可人工阅读和编辑
- 学生更容易理解数据流
- 与实验要求一致

## ADR-002: 模块拆分粒度

**状态**: ✅ 已采纳

**背景**: 实验 2 要求模块化，需要决定拆多少个模块。

**决策**: 拆分为 4 个文件 (model/storage/service/main)，不额外拆分。

**理由**:
- 每个模块职责单一且清晰
- 不增加不必要的抽象层
- 每函数 ≤ 25 行

## ADR-003: done vs delete 实现选择

**状态**: ✅ 已采纳 (后故意引入 bug 再修复)

**背景**: 完成任务操作可以采用标记或删除实现。

**决策**: 使用标记 (`status="done"`)，保留数据完整性。

**理由**:
- 历史记录可追溯
- 按状态过滤有实际意义
- 故意引入 bug (改为 delete) → 测试失败 → 修复

## ADR-004: mock_model 关键词匹配 vs 真实 LLM

**状态**: ✅ 已采纳

**背景**: 实验 8 AI Harness 需要模型输出，但真实 API 调用有成本。

**决策**: 使用关键词匹配的 `mock_model()`，保留替换为真实 LLM API 的接口。

**理由**:
- 零成本、零依赖
- 可复现的确定结果
- 实验 8 明确说"不要求真实调用大模型 API"
